## 5. Semantic findings

### 5a. `onboarding_step_abandoned` IS inflated by backgrounding — CONFIRMED

`Features/Onboarding/OnboardingView.swift:108-116`:

```swift
.onChange(of: scenePhase) { _, newPhase in
    guard newPhase == .background else { return }
    let name = step < Self.analyticsStepNames.count ? Self.analyticsStepNames[step] : "unknown"
    Analytics.trackOnboardingStepAbandoned(
        lastStep: name,
        dwellSeconds: Int(Date().timeIntervalSince(stepEnteredAt))
    )
}
```

Inflated on two independent axes:

1. **Event count.** Fires on *every* transition to `.background`. A user who backgrounds three times on one step emits three `onboarding_step_abandoned` events and may still complete onboarding. It measures backgrounding, not abandonment.
2. **`dwell_seconds` magnitude.** `stepEnteredAt` is reset only on `onAppear` and `.onChange(of: step)` (lines 95-105) — **never on return to foreground**. So `dwell_seconds` measures wall-clock since step *entry*, including all time spent backgrounded. A user who backgrounds for 8 hours and returns emits a second event with `dwell_seconds ≈ 28800` on the same step.

Use diagnostically only. `last_step = 'privacy'` denotes the Coaching UI step (legacy name). Zero production instances to date, so no inflation is yet visible in data — the finding is from code and is structural.

### 5b. `plan_generation_timed_out` duration IS inflated by backgrounding — CONFIRMED

`App/RunSmartLiteAppShell.swift:429-445`:

```swift
private func firstRunnableWorkoutAfterPlanGeneration(timeoutSeconds: TimeInterval = 45) async -> WorkoutSummary? {
    let started = Date()
    let deadline = started.addingTimeInterval(timeoutSeconds)
    while Date() < deadline {
        ...
        try? await Task.sleep(nanoseconds: 500_000_000)
    }
    Analytics.trackPlanGenerationTimedOut(durationMs: Int(Date().timeIntervalSince(started) * 1000))
    return nil
}
```

The loop bound and the reported duration are both **wall-clock `Date()`**, while the work (`nextWorkouts`, `Task.sleep`) is suspended when iOS suspends the app. Backgrounding during polling therefore produces two distortions:

1. **`duration_ms` is inflated without bound.** It reports elapsed wall-clock, not elapsed polling. A 45-second timeout crossed while backgrounded for 10 minutes emits `duration_ms ≈ 600000`. The runbook's "about 45,000" expectation holds only for an uninterrupted foreground poll.
2. **The timeout can fire having barely polled.** On resume, `Date() >= deadline` is already true, so the loop exits immediately and emits `timed_out` even though the plan may exist — a subsequent `plan_generated` / `plan_generation_succeeded` can still arrive. Timeout and success are not mutually exclusive.

Confirming the runbook's caveat: this signals "the activation poll found no runnable workout inside its window", not backend failure. **Do not use `duration_ms` as a backend latency measure.** Zero production instances to date; the finding is structural, from code.

### 5c. NEW — `run_completed` arrives without `run_started` (HealthKit path)

Not in the runbook; surfaced by this snapshot. Build 23 has **2 `run_completed` and 0 `run_started`**. This is not data loss:

- `run_started` has exactly one call site, the in-app run (`Features/Run/RunTabView.swift:253`).
- `run_completed` also fires from HealthKit/Garmin ingestion via `trackCompletedRunIfNeeded` (`Services/Production/RunSmartProductionServices.swift:946`; `Services/Supabase/SupabaseRunSmartServices.swift:1518`), which calls `trackRunCompleted(runType: runType ?? run.source.rawValue, ...)` with **no `run_started`**.

Both observed rows carry `run_type = 'HealthKit'`, consistent with imported runs.

**This independently vindicates the runbook's headline-numerator design.** Had the ordered `windowFunnel` been the decision metric, every HealthKit-imported activation would have been silently erased. The direct `run_completed` numerator is the correct choice.

**But it introduces a numerator-quality caveat E1 must accept.** One observed row:

`run_completed` @ 2026-07-15T17:48:32Z — `distance_km = 0.0`, `duration_s = 101`, `pace_min_km = 0.0`, `run_type = 'HealthKit'`, `is_first_run = **true**`

A 0.0 km, 0.0-pace HealthKit import counted as a **first run**. Under the headline metric ("any `run_completed` within D7"), such a row would count as an activation. `trackCompletedRunIfNeeded` (`AnalyticsEvents.swift:140-158`) applies no distance/duration floor before setting `isFirstRun`. With a real denominator this could inflate the activation numerator. Flagged for the analysis plan; **no code change made or recommended here** (out of scope, and below the evidence threshold to justify one).

## 6. Public sign-in signal (P0, investigate)

Every observed public sign-in attempt on build 23 failed.

| build | tf | emu | `sign_in_completed` | `sign_in_failed` | persons |
|---|---|---|---:|---:|---:|
| 23.0 | False | False | **0** | **5** | 2 |

Across *all* build-23 traffic (physical, TestFlight, emulator), `sign_in_completed` = **0**. Both clean physical entrants (`03766171`, `26140ca4`) failed and neither progressed past sign-in — which fully explains why zero onboarding, plan, or run events exist on the public path.

All 5 failures: `error_domain = 'com.apple.AuthenticationServices.AuthorizationError'`, `error_code = 1000` (`ASAuthorizationError.unknown`).

**These are not user cancellations.** `Features/Auth/SignInView.swift:149-155` explicitly catches and drops `.canceled` (1001) without emitting; only non-cancel errors reach `trackSignInFailed`. Code 1000 is Apple's framework returning *unknown* — the authorization failed before any Supabase token exchange.

**What this is NOT evidence of — stated explicitly:**

- **Not a proven regression.** `trackSignInFailed` was introduced in `5fa2e00` (WP-43/WP-45), which ships *in build 23 itself* (verified via `git log -S`). Earlier builds had no `sign_in_failed` instrumentation, so their clean history is an instrumentation artifact, not evidence they succeeded. There is no valid prior baseline.
- **Not above the interpretation threshold.** n = 2 persons, 5 attempts, ~1 day. The runbook's floor is 10. Two users hitting a transient Apple outage, or two devices in an odd iCloud state, is a live alternative explanation.

What is nonetheless true and worth a founder decision: **no build-23 device has ever completed sign-in**, and the source `com.apple.developer.applesignin` entitlement is present and correct in `RunSmart.entitlements` (so if this is real, the provisioning profile / ASC capability of the shipped binary is the place to look, not the source). Recommend investigating before 2026-07-22 rather than waiting for the cohort to mature — if this is a real auth break, the D7 window is being spent on a funnel nobody can enter.
