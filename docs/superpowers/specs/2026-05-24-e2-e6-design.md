# Spec: E2 + E6 — Weekly Progress Narrative + Post-Run Debrief
*V1.2 · Days 30–90 post-launch · Based on D14 retention signal*
*Created: 2026-05-24*

---

## Problem Statement

Users drop off at D14 because the app goes silent. Two silences:

1. **Between runs** — there is no weekly accumulation of evidence that training is working. Users can't tell if week 2 is better than week 1. No visible progress = no reason to continue.
2. **After each run** — the post-run summary shows stats, but the coach doesn't react. The run feels unwitnessed. No feedback loop = the habit doesn't form.

E2 and E6 together close both silences. E2 makes the week visible. E6 makes each run meaningful.

**Retention link:** both epics address "visible progress" in the retention thesis. Visible progress is one of three conditions for users staying past week 2 (alongside personalization accuracy and safety confidence).

---

## Approach: Sequential (E6 first, then E2)

E6 (post-run debrief) ships in Phase 1. E6 fires on every run and builds a per-run reaction record. By the time E2 (weekly narrative) ships in Phase 2, there is real run-reaction data to synthesize. The weekly narrative is richer because E6 seeded it.

Both features share a single backend architecture: an `intent` field added to the existing `coach_message` Supabase Edge Function.

---

## Section 1: Shared Architecture

### Backend — `coach_message` intent routing

The existing `coach_message` Edge Function handles one intent: conversational `chat`. In V1.2 it gains an `intent` field that routes to different prompt templates.

**Intents after V1.2:**

| Intent | Trigger | Prompt template |
|--------|---------|-----------------|
| `chat` | User sends a message in Coach sheet | Existing conversational prompt (unchanged) |
| `run_debrief` | After each completed run is saved | Run-reaction prompt |
| `weekly_summary` | First app open of each new week | Weekly narrative prompt |

**Request shape extension (backward compatible):**

```typescript
// coach_message/index.ts — new field, defaults to "chat"
intent?: "chat" | "run_debrief" | "weekly_summary"
```

The Edge Function reads `intent` and selects the matching system prompt. Auth, safety rules (`SafetyFlag`), and OpenAI call are unchanged.

**Safety rules carry over unchanged** across all intents:
- No medical advice or diagnosis
- Conservative under uncertainty
- If injury, pain, dizziness, or severe symptom signals appear in context: recommend stop/rest/professional guidance
- No shaming for missed workouts
- No raw GPS coordinates sent to any prompt

### iOS — DTO namespace extension

Two new request/response pairs added to `RunSmartDTO` in `RunSmartAPIModels.swift`:

```swift
// RunSmartDTO namespace additions
struct RunDebriefRequestDTO: Codable { ... }
struct RunDebriefResponseDTO: Codable { ... }
struct WeeklySummaryRequestDTO: Codable { ... }
struct WeeklySummaryResponseDTO: Codable { ... }
```

The existing `CoachMessageRequestDTO` gains an optional `intent: String?` field with a default of `nil` (server defaults to `chat`). No existing call sites change.

## Section 2: E6 — Post-Run Debrief (Phase 1)

### Context

**Current state:** `PostRunLearningCard.swift` is a deterministic view driven by `PostRunLearningCardModel`. It derives "what Coach learned" from local run data with no AI. It already renders in `PostRunSummaryView` and the Garmin/import report path.

**E6 change:** replace the deterministic model with an async AI-generated debrief. Same card surface, smarter content. The deterministic logic becomes the fallback, not the primary.

### Debrief content shape

| Field | Content | Length |
|-------|---------|--------|
| `headline` | Coach reaction to this specific run | 1 line, under 40 chars |
| `debrief` | What was notable, referencing at least one real signal | 1–2 sentences |
| `tomorrow` | What this run means for tomorrow's recommendation | 1 sentence |
| `planImpact` | Optional: plan status after this run | Short phrase |

**Examples:**
- headline: `"Solid — exactly what the plan called for"`
- debrief: `"Your easy pace held steady across all 5km even as HR crept up in the final kilometer. That's aerobic base building working."`
- tomorrow: `"Easy day tomorrow — your legs earned it."`
- planImpact: `"Plan stays on track"`

**Prompt guidance (coach voice):** direct, warm, specific. Not medical. Not generic. Reference the actual run signal (pace vs. target, HR, distance, effort, plan phase). Under 300 characters total for debrief + tomorrow.

### Trigger and loading

Triggered inside `processCompletedActivity` after the run is saved and the activity row exists. The debrief call is async and non-blocking — the post-run summary renders immediately; the debrief card loads separately.

**Loading state:** skeleton placeholder (same card layout, two placeholder lines) while the AI call is in flight. 3-second timeout. On failure or timeout, the existing deterministic content renders instead. The user never sees an error state or loading spinner text.

**Applies to:** RunSmart GPS runs, Garmin imports, and HealthKit imports (wherever `processCompletedActivity` is called).

### Request context (sent to AI)

```swift
struct RunDebriefRequestDTO: Codable {
    var intent: String               // "run_debrief"
    var runDistance: Double          // km
    var runDuration: TimeInterval    // seconds
    var averagePace: Double          // min/km
    var targetPace: Double?          // from planned workout
    var effortRating: Int?           // 1–5 if user rated
    var averageHR: Double?           // bpm if available
    var workoutType: String          // easy / tempo / long / free
    var planPhase: String?           // week N of M
    var recentLoadDays: Int          // runs in last 7 days
    var limitations: [String]        // missing data callouts
}
```

No raw GPS coordinates. No route geometry.

### Response

```swift
struct RunDebriefResponseDTO: Codable {
    var headline: String
    var debrief: String
    var tomorrow: String
    var planImpact: String?
    var safetyFlags: [SafetyFlagDTO]
}
```

## Section 3: E2 — Weekly Progress Narrative (Phase 2)

### Context

E2 ships after E6 is live and generating per-run debriefs. By Phase 2, there is at least one full week of run data (and debrief reactions) to synthesize into a meaningful weekly narrative.

### Weekly card content shape

| Field | Content | Length |
|-------|---------|--------|
| `headline` | One stat that proves something changed | 1 line, under 50 chars |
| `narrative` | Coach voice summary of the week | 2–3 sentences |
| `forwardLook` | What next week is building toward | 1 sentence |
| `weekLabel` | Context label for the card | "Week 4 of your plan" / "Week 3 with RunSmart" |

**Examples:**
- headline: `"3 runs · 18 km · 4th week in a row"`
- narrative: `"You held easy pace across all three runs this week even as total distance stepped up 15%. Your aerobic base is absorbing the load. That's exactly what week 4 should feel like."`
- forwardLook: `"Next week's long run is where this base starts to pay off."`
- weekLabel: `"Week 4 of your plan"`

**Prompt guidance:** coach voice, specific to this runner's actual week, non-shaming about missed workouts ("3 of 4 planned runs — life happens, the trend is good"), forward-looking. Under 400 characters total for narrative + forwardLook.

### Trigger

First app open of each new week (calendar week boundary: Monday 00:00 local time). The weekly summary is generated once and cached for the full week. If the user opens on Tuesday, they see Monday's narrative unchanged. The next Monday it regenerates.

**Generation guard:** only generate if at least one run was completed in the past 7 days. If no runs, the card does not appear (no empty "you did nothing this week" card).

### Card placement on Today

Card slot: below `TodayRecommendationCard`, above `Beginner5KHabitCard`. Only shows when a weekly summary exists for the current week.

Tapping the card opens the Coach sheet with the weekly context pre-loaded, so the user can ask follow-up questions ("Why was this week a step-up?", "Will next week be harder?").

### Request context (sent to AI)

```swift
struct WeeklySummaryRequestDTO: Codable {
    var intent: String               // "weekly_summary"
    var weekNumber: Int              // week N of the active plan
    var weekStartDate: String        // ISO date
    var runsCompleted: Int
    var runsPlanned: Int
    var totalDistanceKm: Double
    var avgPaceMinPerKm: Double?
    var prevWeekDistanceKm: Double?  // for step-up/step-back context
    var planPhase: String?           // build / peak / taper / base
    var isRecoveryWeek: Bool
    var readinessAverage: Double?    // 0–100 if available
    var limitations: [String]
}
```

No raw GPS. No route geometry.

### Response

```swift
struct WeeklySummaryResponseDTO: Codable {
    var headline: String
    var narrative: String
    var forwardLook: String
    var weekLabel: String
    var safetyFlags: [SafetyFlagDTO]
}
```

### Fallback

If AI generation fails: a deterministic card renders with `headline` only (runs + distance) and a static phrase: `"A solid week of training."` No error copy. The card still appears and still proves something.

## Non-Goals (V1.2)

- No push notifications for E2 or E6 (in-app only; push is a later enhancement)
- No user feedback or thumbs up/down on debrief or narrative
- No weekly progress history screen (card is current week only)
- No E2 card on Plan tab (Today only)
- No debrief for rest days or planned skips
- No E5 (Adaptive Plan / Flex Week) in this spec — separate epic

---

## Decisions (2026-05-24)

- **E6 debrief storage:** store in Supabase alongside the activity row. Survives device reinstalls and is available for E2 weekly narrative synthesis. Requires a new `run_debriefs` table or a `debrief` JSONB column on the existing activities table.
- **Week boundary locale:** use `Calendar.current.firstWeekday` per device locale. The weekly card triggers on the first day of the week as the device calendar defines it (Sunday-start devices trigger Sunday; Monday-start devices trigger Monday).
- **E4 + E2 coexistence:** when a user is on the 21-Day Rookie Challenge (`Beginner5KHabitCard` is active), the `WeeklyProgressCard` is suppressed. The habit card takes priority on Today during the challenge. `WeeklyProgressCard` becomes visible after the challenge completes or lapses.
